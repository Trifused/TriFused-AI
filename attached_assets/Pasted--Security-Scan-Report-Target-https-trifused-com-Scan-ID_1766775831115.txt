### 2. Missing Permissions-Policy Header

**Type:** `missing-permissions-policy`
**Source:** http-analyzer
**Confidence:** high
**Affected URL:** `https://trifused.com/`
**Affected Component:** HTTP Headers

#### Problem

Missing Permissions-Policy header. Sensitive browser features (camera, microphone, geolocation) may be accessible to embedded third-party content.

#### How to Fix

## Fix
Add Permissions-Policy header to your server configuration.

## Before applying
Check if this application uses any of these browser APIs:
- Geolocation API (navigator.geolocation)
- Camera/Microphone (getUserMedia, WebRTC)
- Payment Request API
- Fullscreen API

## Implementation

**If the app does NOT use camera/mic/geolocation:**
```
Permissions-Policy: geolocation=(), microphone=(), camera=()
```

**If the app DOES use these features (allow for same origin only):**
```
Permissions-Policy: geolocation=(self), microphone=(self), camera=(self)
```

**If third-party embeds need access:**
```
Permissions-Policy: camera=(self "https://trusted-domain.com")
```

## Warning
Setting `=()` completely DISABLES that browser feature. If the application uses geolocation/camera/microphone, use `=(self)` instead or the feature will break.

#### Evidence

**Missing Header:** `Permissions-Policy`

### 3. Missing Referrer-Policy Header

**Type:** `missing-referrer-policy`
**Source:** http-analyzer
**Confidence:** high
**Affected URL:** `https://trifused.com/`
**Affected Component:** HTTP Headers

#### Problem

Missing Referrer-Policy header. The Referer header may leak sensitive information (URLs with tokens, session IDs) to third-party sites when users click external links.

#### How to Fix

## Fix
Add Referrer-Policy header to prevent information leakage:

```
Referrer-Policy: strict-origin-when-cross-origin
```

## Recommended Values

**Most secure (recommended for apps with sensitive URLs):**
```
Referrer-Policy: no-referrer
```
- No referrer information sent at all
- Use if your URLs contain tokens, session IDs, or sensitive data

**Balanced (recommended for most apps):**
```
Referrer-Policy: strict-origin-when-cross-origin
```
- Sends full URL for same-origin requests
- Only sends origin for cross-origin requests
- Degrades to origin-only on HTTPSâ†’HTTP

**Less restrictive:**
```
Referrer-Policy: same-origin
```
- Only sends referrer for same-origin requests
- No referrer for cross-origin requests

## Safe to apply
This header is safe to add. If you use analytics that rely on referrer data, use `strict-origin-when-cross-origin` instead of `no-referrer`.

#### Evidence

**Missing Header:** `Referrer-Policy`

### 5. Missing Cross-Origin-Opener-Policy Header

**Type:** `missing-coop`
**Source:** http-analyzer
**Confidence:** high
**Affected URL:** `https://trifused.com/`
**Affected Component:** HTTP Headers

#### Problem

Missing Cross-Origin-Opener-Policy (COOP) header. Without this, cross-origin windows can maintain references to your window object.

#### How to Fix

## Fix
Add COOP header to your server configuration.

## Before applying
Check if this application uses any popup-based flows:
- OAuth/SSO login popups (Google, GitHub, etc.)
- Payment popups (Stripe, PayPal)
- `window.open()` calls that need to communicate back to the opener

Search codebase for: `window.open`, `window.opener`, OAuth callback handling

## Implementation

**If NO popup flows are used:**
```
Cross-Origin-Opener-Policy: same-origin
```

**If OAuth or payment popups ARE used:**
```
Cross-Origin-Opener-Policy: same-origin-allow-popups
```

## Do NOT apply `same-origin` if
The app uses OAuth popup flows or payment provider popups - it will break authentication/checkout.

#### Evidence

**Missing Header:** `Cross-Origin-Opener-Policy`

### 1. CSP allows 'unsafe-eval'

**Type:** `weak-csp`
**Source:** http-analyzer
**Confidence:** high
**Affected URL:** `https://trifused.com/`
**Affected Component:** Content-Security-Policy

#### Problem

The Content-Security-Policy includes 'unsafe-eval' in script-src, which allows use of eval(), Function(), and similar methods.

Note: Many frameworks require 'unsafe-eval' to function (Vue.js templates, certain bundlers, Stripe.js, Angular in development mode). If you cannot remove it, ensure other CSP directives are strict and input validation is robust.

#### How to Fix

If possible, remove 'unsafe-eval' from script-src. Refactor code to avoid:
- eval()
- Function() constructor
- setTimeout/setInterval with strings

However, if your framework requires it (Vue.js, Stripe, some webpack configs), focus on:
1. Implementing nonce-based CSP for inline scripts
2. Ensuring other directives are strict
3. Implementing robust input validation

#### Evidence

  "csp": "default-src 'self'; script-src 'self' 'unsafe-inline' 'unsafe-eval' https://www.googletagmanager.com https://www.google-analytics.com; style-src 'self' 'unsafe-inline' https://fonts.googleapis.com; font-src 'self' https://fonts.gstatic.com; img-src 'self' data: https: blob:; connect-src 'self' https://www.google-analytics.com https://pagespeedonline.googleapis.com https://*.googleapis.com; frame-ancestors 'self';",
